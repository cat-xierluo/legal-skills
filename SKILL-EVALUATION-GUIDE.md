# Skill 评估指南（内部草案）

本指南是 [SKILL-DEV-GUIDE.md](SKILL-DEV-GUIDE.md)、[SKILL-ORCHESTRATION-GUIDE.md](SKILL-ORCHESTRATION-GUIDE.md) 与 [SKILL-HANDOFF-GUIDE.md](SKILL-HANDOFF-GUIDE.md) 的补充文档。

- **SKILL-DEV-GUIDE.md**：单个 Skill 的开发规范
- **SKILL-ORCHESTRATION-GUIDE.md**：多个 Skill 的协作编排规范
- **SKILL-HANDOFF-GUIDE.md**：多个 Skill 之间的交接契约规范
- **SKILL-EVALUATION-GUIDE.md**：Skill 的评估方法、验收分层与迭代框架

---

## 1. 背景

官方新版 `skill-creator` 已经明显把重心从“如何写一个 Skill”推进到“如何验证一个 Skill 是否真的可用、可泛化、可维护”。

对 `legal-skills` 而言，这个变化尤其重要。法律类 Skill 的问题往往不在格式，而在以下方面：

- 表面结构正确，但事实提取不准
- 输出看起来专业，但没有严格依据
- 遇到信息不足时没有先追问，反而直接下结论
- 风险分级、服务边界、免责声明口径不稳定
- 在少量案例上表现良好，但换一个材料类型就失稳

因此，法律类 Skill 的质量控制不能只依赖：

- `lint` 式静态检查
- 一次性的人工试跑
- “看起来差不多”的主观印象

需要单独建立一套面向法律业务的评估框架。

## 2. 目标

本框架的目标不是消灭人工复核，而是让人工复核更聚焦、更可复用、更可回归。

本框架要解决四个问题：

1. 区分“格式合规”与“任务效果”
2. 让复杂法律 Skill 具备可比较、可复测、可迭代的评估基础
3. 将人工复核从“每次从头判断”变成“按统一标准审看”
4. 为后续的 `skill-architect`、`skill-lint`、专门 eval skill 提供统一语言

## 3. 基本判断

### 3.1 `lint` 不等于 `eval`

`lint` 负责检查：

- 目录结构
- frontmatter
- 引用路径
- 文档一致性
- 是否存在 benchmark、rubric、输出模板等评估基础设施

`eval` 负责检查：

- 技能是否在真实任务中做对了事
- 是否在输入不足、材料混乱、边界模糊时仍能稳定工作
- 是否遵守法律服务边界与安全要求

### 3.2 `quick_validate` 不等于法律质量验证

官方的 `quick_validate` 更接近“Skill 文件夹合法性检查”，不是对法律分析质量的判定。

对于法律类 Skill，真正重要的问题是：

- 是否忠于材料
- 是否清晰区分事实、判断、建议
- 是否明确标注信息不足
- 是否引用了正确的依据
- 是否控制了过度承诺和法律风险

### 3.3 评估的目标是“泛化”，不是“复现预期答案”

评估时应尽量避免泄漏标准答案、预设结论或意图修复点。

重点应放在：

- 给原始材料
- 给接近真实用户的话术
- 看 Skill 是否能自己重建问题空间

如果一个 Skill 只有在测试者知道“正确答案是什么”的情况下才能通过，那么这个 Skill 还不够稳。

## 4. 适用范围

不是所有 Skill 都需要同等强度的 eval。

### 4.1 低强度评估：低自由度、结果较确定

适用类型：

- 文本格式化
- 内容搬运与转换
- 固定模板生成
- 以脚本执行为主的工具类 Skill

代表示例：

- `legal-text-format`
- `md2word`
- `mineru-ocr`

主要关注：

- 输出完整性
- 格式稳定性
- 输入输出一致性
- 是否破坏原始内容

### 4.2 中强度评估：结构化专业输出

适用类型：

- 风险初筛
- 类别规划
- 问答提取
- 规则式分析报告

代表示例：

- `trademark-assistant`
- `legal-qa-extractor`

主要关注：

- 是否先收集必要信息
- 是否按标准结构输出
- 是否稳定执行风险分级
- 是否遵守边界与免责声明要求

### 4.3 高强度评估：高自由度法律分析与策略判断

适用类型：

- 诉讼分析
- 专利侵权/稳定性/FTO 分析
- 法律方案生成
- 需要持续迭代提示词与参考材料的复杂分析类 Skill

代表示例：

- `litigation-analysis`
- `patent-analysis`
- `legal-proposal-generator`

主要关注：

- 事实抽取是否准确
- 法律推理是否可追溯
- 是否明确不确定性与信息缺口
- 是否真正帮助决策，而不是只写出“像样”的报告

## 5. 五层评估对象

对法律类 Skill，建议至少按五层来设计评估。

### 5.1 Trigger Eval

评估问题：

- 这个 Skill 是否应该在该请求中触发
- 是否越过了自身边界
- 是否误处理了本应由其他 Skill 处理的任务

典型失败：

- 把“正式法律意见”当成初筛任务处理
- 明明应该先抓取或 OCR，却直接进入法律分析

### 5.2 Intake Eval

评估问题：

- 是否识别了关键信息缺失
- 是否在信息不足时先追问
- 是否知道哪些缺失项会阻断有效判断

典型失败：

- 缺少商标使用场景仍直接评级
- 缺少案件关键事实仍直接给出胜诉判断

### 5.3 Reasoning Eval

评估问题：

- 事实提取是否准确
- 是否区分“材料原文”与“模型归纳”
- 法律依据是否匹配结论
- 推理链条是否自洽

典型失败：

- 编造材料中不存在的关键事实
- 结论与引用依据脱节
- 把假设性内容写成既定事实

### 5.4 Output Eval

评估问题：

- 输出结构是否完整
- 面向对象是否正确
- 风险等级、建议动作、下一步是否清楚
- 是否便于后续复用或人工接手

典型失败：

- 结构齐全但不能用于真实交付
- 结论很多，但没有明确行动建议

### 5.5 Safety Eval

评估问题：

- 是否泄露个人/企业敏感信息
- 是否过度承诺结果
- 是否跨法域错误适用规则
- 是否缺少免责声明、升级建议、律师介入提示

典型失败：

- 用初筛语气承诺注册成功率
- 未脱敏就提取问答对
- 未说明适用法域限制

## 6. Hard Fail 与 Rubric

### 6.1 先判 Hard Fail，再做细分评分

法律类 Skill 不适合一开始就追求一个“总分”。

建议先判断是否出现 Hard Fail。只要出现任一 Hard Fail，本次评估即不能视为通过。

### 6.2 建议的 Hard Fail 条件

以下情况建议统一视为 Hard Fail：

- 编造关键事实、证据、时间、主体或结论依据
- 未标注信息不足，却给出确定性法律结论
- 泄露应脱敏信息
- 将初筛/辅助分析表述为正式法律意见
- 未按 Skill 规则附加必要免责声明或升级建议
- 引用明显错误或与结论不相关的法律依据
- 在高风险场景下没有提示人工律师介入

### 6.3 建议的细分评分维度

在未触发 Hard Fail 的前提下，可按 0-2 分评分。

| 维度 | 0 分 | 1 分 | 2 分 |
|------|------|------|------|
| 触发与边界 | 明显误触发/越界 | 基本正确但边界含糊 | 触发准确且边界清晰 |
| 输入收集 | 未识别缺失信息 | 识别部分缺失 | 能主动识别并追问关键缺口 |
| 事实忠实度 | 有明显错漏 | 基本准确 | 准确且区分原文与归纳 |
| 法律推理 | 推理混乱或错配 | 推理基本成立 | 推理清楚且依据匹配 |
| 输出可用性 | 难以交付/复用 | 基本可用 | 结构清晰且可直接接手 |
| 安全与合规 | 存在明显风险 | 有轻微风险 | 边界、脱敏、免责均稳定 |

说明：

- 不建议把总分神化为唯一标准
- 复杂 Skill 更应关注“哪一层失效”
- 评估记录应保留定性意见，而不是只留分数

## 7. Benchmark Case 的最小结构

每个需要持续迭代的法律 Skill，建议逐步建立 benchmark case 集。

### 7.1 单个 case 建议包含

1. `case_id`
2. 用户原话
3. 原始材料路径或摘录
4. 任务类型
5. 关键输入缺口
6. 必须包含项
7. 禁止项
8. Hard Fail 判定条件
9. 人工评语模板

### 7.2 示例模板

```markdown
# CASE: ta-001

## 用户原话
请帮我看一下这个商标大概能不能注册，顺便给我建议注册哪些类别。

## 输入材料
- 商标名称：XX
- 主营业务：咖啡与零售
- 缺失：具体商品/服务清单、是否已有图形方案

## 任务类型
- 类别规划
- 可注册性初筛

## 关键输入缺口
- 实际使用商品/服务范围
- 是否为文字商标、图形商标或组合商标

## 必须包含项
- 明确提示信息不足
- 在类别规划与可注册性初筛之间做区分
- 给出下一步补充信息建议
- 附带免责声明与升级建议

## 禁止项
- 直接承诺可注册
- 在信息不足时给出确定性低风险判断

## Hard Fail
- 未提示信息不足
- 承诺注册成功率
```

## 8. 评估执行流程

建议将评估分成四步，而不是一次性混在一起做。

### 8.1 第一步：静态检查

目标：

- Skill 结构是否合法
- 是否具备最基础的说明、模板、引用、边界表达
- 是否已经声明输入输出与限制

执行者：

- `skill-lint`

### 8.2 第二步：基础验证

目标：

- 规则是否自洽
- 关键模板是否存在
- 如果有脚本，脚本能否运行

执行者：

- `quick_validate`
- 手工检查
- 必要时脚本自测

### 8.3 第三步：Benchmark Eval

目标：

- 在已知样本上观察 Skill 是否稳定
- 找出最常见的失效模式

执行者：

- 专门的 eval skill
- 人工复核

### 8.4 第四步：Forward-testing

目标：

- 测试 Skill 在接近真实场景中的泛化能力
- 检查没有标准答案时能否仍然稳定工作

执行原则：

- 使用尽量少泄漏上下文的提示
- 优先给原始材料，不给“正确答案”
- 使用 fresh thread / 独立测试轮次
- 每次迭代后重新从原始材料出发

## 9. 人工复核的角色

人工复核不应被视为“eval 失败的替代品”，而应被视为框架中的正式一层。

建议人工复核重点看：

- Hard Fail 是否发生
- 输出是否能进入真实工作流
- 哪些问题是规则缺失，哪些问题是执行不稳
- 是否值得把本次经验沉淀成新的 benchmark case

人工复核不需要每次都从头自由判断。应尽量按统一 checklist 审看。

## 10. 三类工具的职责分工

### 10.1 `skill-architect`

后续负责回答：

- 创建或改造一个法律 Skill 时，必须提前设计哪些 eval 要素
- 哪类 Skill 需要哪种强度的 benchmark 与人工复核
- 如何定义输入、输出、边界、Hard Fail

### 10.2 `skill-lint`

后续负责回答：

- 这个 Skill 是否“具备可评估性”

建议未来新增的检查项包括：

- 是否声明评估范围
- 是否声明 Hard Fail
- 是否提供 benchmark case 或样例
- 是否有输出验收标准
- 是否区分静态检查与动态评估

### 10.3 专门的 eval skill

后续负责回答：

- 这个 Skill 在具体任务上是否做对了
- 失败发生在哪一层
- 本轮修改是否比上一轮更稳

## 11. 对现有法律 Skill 的落地建议

建议分三批推进，不要全仓库同时改。

### 第一批：低自由度试点

- `legal-text-format`

目标：

- 先跑通“完整性验证 + benchmark case + 人工复核模板”

### 第二批：中自由度试点

- `trademark-assistant`
- `legal-qa-extractor`

目标：

- 固化“信息不足先追问”
- 固化“边界、免责声明、升级建议”
- 建立第一版风险分级验收标准

### 第三批：高自由度试点

- `litigation-analysis`
- `patent-analysis`

目标：

- 拆出多层 rubric
- 建立高风险场景 Hard Fail
- 将“专业性”改写为可观察的验收项

## 12. 当前版本的结论

当前阶段最合理的策略不是立刻重写所有 Skill，而是：

1. 先统一评估语言
2. 再决定如何改 `skill-architect`
3. 再决定如何改 `skill-lint`
4. 最后逐个 Skill 补 benchmark、rubric 和回归机制

换句话说，下一步不是问“这个 Skill 写得像不像”，而是问：

**这个 Skill 是否已经被设计成可评估、可复核、可迭代的法律工作单元。**
