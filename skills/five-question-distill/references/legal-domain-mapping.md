# 法律领域识别与模板映射

本文档定义如何从素材或场景描述中识别法律领域，并映射到对应的 prompt 模板骨架。对应 SKILL.md 步骤 1A.2 与步骤 2。

## 一、识别信号

### 从素材中识别

| 信号词 | 倾向领域 |
|--------|---------|
| 起诉状、答辩状、判决书、庭审、立案、上诉 | 诉讼 |
| 合同、协议、条款、审查意见、谈判 | 合同 |
| 合规、内控、风控、整改、监管 | 合规 |
| 尽调、尽职调查、并购、收购、DD | 尽调 |
| 商标、专利、著作权、商业秘密、侵权 | 知产 |
| 解除劳动合同、工伤、仲裁、工资、社保 | 劳动 |
| 股权、股东、章程、决议、增资 | 公司 |
| 遗嘱、继承、离婚、抚养、财产分割 | 家事 |
| 发票、纳税、税务稽查、增值税 | 税务 |
| 刑事、侦查、取保、辩护、罪名 | 刑事 |

### 从场景描述中识别

| 用户描述 | 倾向领域 |
|---------|---------|
| "我常给企业做合规体检" | 合规 |
| "我做知识产权维权" | 知产 |
| "我代理劳动争议原告" | 诉讼（劳动） |
| "我审合同审得多了" | 合同 |

## 二、主分类 vs 辅分类

很多 Skill 跨多个领域（如"劳动合同审查"= 合同 + 劳动）。规则：

- **主分类**：决定 prompt 的主结构骨架。选动作类型最重的那一类。
- **辅分类**：作为 knowledge_base 的领域规则包注入。

判断"主"的标准：看用户最频繁做的动作。审查是动作 → 主分类"合同"；劳动是被审对象 → 辅分类"劳动"。

## 三、模板骨架映射

每个主分类对应一份 prompt 骨架（详见 standard-prompt-output.md 的模板填充示例）：

| 主分类 | workflow 骨架典型形态 |
|--------|---------------------|
| 合同 | 接收合同 → 分类（类型/风险等级）→ 逐条审查 → 风险清单 → 修改建议 |
| 诉讼 | 接收案情 → 法律关系识别 → 诉讼请求设计 → 证据组织 → 文书起草 |
| 合规 | 接收企业信息 → 风险扫描 → 缺口识别 → 整改建议 → 持续监控 |
| 尽调 | 接收目标 → 事项清单 → 资料收集 → 风险标记 → 报告生成 |
| 知产 | 接收权利基础 → 侵权比对 → 维权路径选择 → 文书起草 |

## 四、识别不确定时的处理

如果素材或描述模糊，**不要硬猜**：

1. 向用户提供 2-3 个候选分类，让用户选。
2. 如果用户自己也不确定，先用"通用法律文书"骨架起步，等五问访谈完再回填具体领域。

宁可起步用通用骨架，也不要错分领域导致模板选错。
