# 五问蒸馏法 Skill

## 概述

把法律人零散的"经验/素材/直觉"转化为一份结构化、可被 `legal-skill-creator` 直接编译的**标准 prompt**。这是写 Skill 前的"前置蒸馏器"——不直接生成 Skill 文件，只产出喂给 skill creator 的输入。

本 Skill 对应本书 ch07《单个 Skill 的编写四步法》第二步：原"三问"（输入/输出/处理逻辑）补全为"五问"。

**核心价值**：
- **双路径覆盖**：用户有素材就走蒸馏路径，没素材就走访谈路径，不假设用户已经准备好。
- **五问齐全**：确保产出的 prompt 五要素齐全，下游 skill creator 不会因为缺要素而返工。
- **法律分类先行**：先识别法律领域，再选模板，避免"通用模板套法律场景"的失真。

## 五问是什么（必背）

| # | 要素 | 解决的问题 | 在 prompt 中对应的字段 |
|---|------|-----------|----------------------|
| 1 | **输入什么** | 这个 Skill 吃什么数据 | `inputs` |
| 2 | **输出什么** | 产出长什么样 | `outputs` |
| 3 | **处理逻辑** | 从输入到输出怎么走 | `workflow` |
| 4 | **向谁交付** | 终端用户是谁、什么场景触发 | `context.party_role` / `context.trigger` |
| 5 | **需要哪些知识支撑** | 法律依据、范本、风险清单、风格偏好 | `knowledge_base` / `references` |

第 4、5 问是本书相对"经典三问"补全的两问——**向谁交付**决定语气与详略，**需要哪些知识支撑**决定这个 Skill 是"空壳模板"还是"有肉的真本事"。

## 工作流程

### 步骤 0：判定路径（最先做）

读用户第一条消息，判断走哪条路径：

| 用户说话特征 | 路径 |
|------------|------|
| 提到"这是我的样本/SOP/Q&A/历史文书/案例"，或粘贴/上传了素材 | **有素材路径** |
| 说"我想做个 XX skill 但不知道怎么开始""我脑子里有经验但没整理过""教我做" | **无素材路径** |
| 同时满足 | 先走有素材路径，再用访谈补缺口 |

不要在没素材的情况下硬走蒸馏路径——会产出空洞的 prompt。

### 步骤 1A：有素材路径——三分类蒸馏

#### 1A.1 材料三分类（必做）

把用户给的所有素材归入三类（可多类共存）：

| 分类 | 识别特征 | 蒸馏重点 |
|------|---------|---------|
| **样本（Sample）** | 成品文书：律师函、起诉状、合同、审查意见 | 提炼结构骨架 + 风格特征 + 风险点清单 |
| **流程 SOP** | 步骤化文档：办案流程、审查清单、出庭准备 | 提炼决策点 + 分支条件 + 质量检查点 |
| **问答 Q&A** | 一问一答：客户咨询、培训答疑、内部 FAQ | 提炼高频问题 + 标准答法 + 例外情形 |

蒸馏策略详见 [distill-strategies.md](references/distill-strategies.md)。

#### 1A.2 法律领域识别

从素材中识别法律领域，影响后续 prompt 模板选择：

- **合同**：起草、审查、谈判、违约处理
- **诉讼**：起诉、答辩、上诉、再审、执行
- **合规**：企业合规、行业合规、数据合规、反舞弊
- **尽调**：法律尽调、财务尽调、并购尽调
- **知产**：商标、专利、著作权、商业秘密
- **其他**：劳动、公司、税务、家事、刑事等

映射规则详见 [legal-domain-mapping.md](references/legal-domain-mapping.md)。

#### 1A.3 引导补全五问

蒸馏完，把已从素材中提取的要素列给用户看，**明确指出缺口**（通常是第 4 问"向谁交付"和第 5 问"知识支撑"），引导用户补全。一次只问 1-2 个缺口，不要一次抛出五个空字段吓退用户。

### 步骤 1B：无素材路径——苏格拉底式访谈

不假设用户有任何素材。通过逐题引导，从用户脑中的"经验直觉"挖出五问答案。

访谈遵循三条铁律：

1. **一次一问**：不要一次列五个问题，用户会放弃。
2. **给锚点不给答案**：每个问题配 2-3 个具体例子帮助用户联想，但不替用户做决定。
3. **追问直到具体**：用户说"帮助客户"→ 追问"哪个阶段、什么角色、什么场景下需要帮助"。

访谈问题清单详见 [interview-guide.md](references/interview-guide.md)。

### 步骤 2：识别法律分类（两条路径都要做）

无论素材路径还是访谈路径，最终都要落到一个**法律分类**上。这个分类决定 prompt 模板的骨架。识别方法：

- 有素材：从素材的文书类型、案件类型推断（步骤 1A.2）
- 无素材：在访谈第 1 问"你平时重复做的那件事是什么"时识别

### 步骤 3：产出标准 prompt

把五问的答案组装成一份**标准 prompt**，格式能让 `legal-skill-creator` 直接消费。

输出模板详见 [standard-prompt-output.md](references/standard-prompt-output.md)。

**关键约束**：
- prompt 必须包含 `skill_family`、`jurisdiction`、`stage`、`party_role`、`doc_type` 五个必填标识符（对齐 legal-skill-creator 第二章"向用户确认的信息"）。
- 五问要素必须齐全，缺一不可。如果某要素访谈完仍无法确定，标注 `[待用户确认]` 并在 prompt 末尾列"待确认清单"，**不要编造**。

### 步骤 4：交付与移交

1. 把标准 prompt 展示给用户确认。
2. 用户确认后，**明确提示**下一步应调用 `legal-skill-creator`（或通用 `skill-creator`）来编译出真正的 Skill 文件。
3. 本 Skill 不直接产出 SKILL.md / references / scripts——那是下游的事。

## 五问要素自检清单

产出 prompt 前，**必须**逐条核对：

- [ ] 第 1 问 输入什么：列出了必要输入与可选输入，标注了信息来源
- [ ] 第 2 问 输出什么：明确了产出格式、字数/结构要求、质量标准
- [ ] 第 3 问 处理逻辑：写了主要步骤、决策点、可选分支
- [ ] 第 4 问 向谁交付：明确了终端用户角色、触发场景、语气基调
- [ ] 第 5 问 知识支撑：列出了法律依据、范本、风险清单、风格偏好的来源

任一项缺失或模糊，回到对应步骤补全，**不放过半成品**。

## 输入要求

本 Skill 接受以下输入：

1. **素材路径**：用户提供的文件路径（支持 .md/.txt/.docx）
2. **粘贴内容**：用户直接粘贴的样本/SOP/Q&A 内容
3. **场景描述**：用户用自然语言描述"我想做一个 XX skill"
4. **混合**：上述任意组合

## 输出要求

- 一份结构化的标准 prompt（markdown 格式）
- 末尾附"待确认清单"（如有）
- 明确提示调用 `legal-skill-creator` 完成编译

## 适用场景

- 律师把多年积累的某类文书做成可复用 Skill
- 律所把内部 SOP 沉淀为团队级 Skill
- 从客户咨询记录中提炼可复用问答 Skill
- 法律培训师把课程方法论转化为 Skill
- 任何"我有经验/素材，想做成 Skill 但不知如何下手"的场景

## 边界

- **不直接生成 Skill 文件**：只产出 prompt，编译交给 `legal-skill-creator`。
- **不替代法律判断**：蒸馏出的"处理逻辑"和"风险清单"需由执业律师最终确认。
- **不做证据不足的推断**：素材或访谈不足以支撑的结论，标注 `[需补充]`，不臆测。
