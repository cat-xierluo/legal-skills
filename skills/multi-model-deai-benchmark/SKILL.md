# Multi-Model De-AI Benchmark

## 定位

同一章书稿让不同模型跑，AI 味轻重差别很大。本 skill 不重新发明去 AI 化检测逻辑，而是**对接既有规范**：

- 评价标准来源 ①：本书 `docs/BOOK-TONE-REVIEW.md` 的七类口吻门禁。
- 评价标准来源 ②：`de-ai-polish` skill 的检测方法（频次红线、向上溯源、结构机械感）。

裁判模型按上面这两套既有规范给每份候选输出打分，排出名次，然后建议用哪个模型跑定稿。

## 适用与不适用

**适用**：
- 同一写作任务，想横向对比 2~5 个候选模型的去 AI 化表现。
- 想沉淀"哪个模型在法律书稿去 AI 化上最稳"的经验（选型记录）。
- 已经有候选输出，想让裁判模型给一份结构化排名。

**不适用**：
- 只跑一个模型、单点去 AI 化（直接用 `de-ai-polish`）。
- 想自动选模型 API / Provider 配置（用 `myagents-cli` 的 model 能力）。
- 想审稿维度更广（术语、跨章一致性等，用 `writing-reviewer`）。

## 核心工作流程

### Step 0：对齐输入（必做，少于 1 分钟）

确认三件事，缺一不可：

1. **写作任务定义**：要写的是哪一段 / 哪一章、约多少字、什么文体（导语 / 正文 / 案例框 / 操作步骤）。最好附带 prompt 原文，让每个模型用完全一样的输入。
2. **候选模型清单**：N 个模型（建议 2~5 个），写清每个模型的口径（如 "Claude Sonnet 直连" vs "Qoder 走 Claude 旗舰风格" 要分开记，因为它们风格不同）。
3. **裁判模型**：默认用一个**与候选模型不同家族的强模型**当裁判（如候选都是国产模型，裁判用 GPT / Claude 旗舰）。裁判必须**不在候选清单里**，避免自评偏差。

> 没有这三项时，不要硬跑。先问用户补齐，特别是裁判模型不能缺席。

### Step 1：并行生成 N 份候选

用**完全相同的 prompt** 让 N 个候选模型各跑一遍，输出 N 份草稿。建议：

- 每份草稿单独存盘，文件名带模型标识：`draft-glm.md`、`draft-claude.md`、`draft-qoder-claude.md`、`draft-deepseek.md`。
- 不要让模型在生成阶段做去 AI 化（否则比的是"谁的内置 polish 强"，不是"谁原生 AI 味轻"）。prompt 里只给写作任务本身。
- 如果模型本身就会自动去 AI 化（如某些旗舰模型默认带 polish），在记录里标出来，但仍然按"原生输出"参评。

### Step 2：裁判评价（核心步骤）

把 N 份草稿**同时**交给裁判模型，让它按 `references/judge-prompt.md` 的模板打分。

裁判要做的事：

1. **逐份扫七类口吻门禁**（BOOK-TONE-REVIEW）：调研报告口吻 / 写作取舍口吻 / 模板化对比句 / 空泛意义拔高 / 工整排比 / 框架提示句与碎句感 / 高频模糊词。每一类命中几处、命中在哪段，写出来。
2. **逐份跑 de-ai-polish 检测维度**：频次红线词（越来越 / 其实 / 往往 / 真正 等，全文超过 2 次即记）、向上溯源（未列出的变体是否属于已知模式家族）、结构机械感（三并列 / 程式化引导词）。
3. **打分**：每份按 `references/scoring-rubric.md` 的五维（口吻干净度 / 句式自然度 / 词汇准确度 / 信息密度 / 法律边界感）各打 1-5 分，给出加权总分。
4. **横向排名**：N 份草稿从高到低排，每份附 2-3 条最致命的问题 + 1-2 条独特优势。

> 裁判 prompt 模板见 `references/judge-prompt.md`，评分维度见 `references/scoring-rubric.md`。这两个文件**对接既有规范**，不另起炉灶。

### Step 3：选优 + 定稿

根据裁判排名：

1. **首选**：总分最高且没有致命口吻问题的模型，用来跑定稿。
2. **首选不稳时的回退策略**：
   - 如果第一名只是某一项弱（如句式自然度低但口吻干净），可以**第一名跑初稿 + 用 de-ai-polish 单点修弱项**，而不是换模型重跑。
   - 如果前两名分数接近（差距 ≤ 0.5 分），再各跑一段相同样量的不同段落，做第二轮 tiebreak。
3. **定稿**：选定模型后，按本书正常写作流程跑定稿；定稿出来后**仍然要走一次 de-ai-polish 和 BOOK-TONE-REVIEW 门禁**，横评只解决"选哪个模型"，不替代终检。

### Step 4：沉淀选型经验（每次必做）

把本次横评结果记到 `references/model-selection-log.md`：

- 写作任务类型（导语 / 正文 / 案例框 / 操作步骤）。
- 参评模型清单 + 裁判模型。
- 排名 + 每份致命问题。
- 最终选用哪个模型、为什么（含回退策略）。
- 一句话结论（如 "本章正文 Claude 旗舰风格最稳，Qoder-Claude 次之；案例框 DeepSeek 更具体"）。

> 这个日志是本 skill 的核心沉淀。多次积累后，能直接看出"哪类写作任务该用哪个模型"，不用每次都全量横评。

## 最优模型选型经验（初始版，随日志更新）

基于本书前期试跑的经验，**初始经验**如下（仅供第一轮横评参考，最终以 `model-selection-log.md` 实际记录为准）：

| 写作任务 | 首选模型 | 备选 | 说明 |
|---------|---------|------|------|
| 章节正文（论证 / 论述型） | Claude 旗舰风格（直连或 Qoder 接入） | GLM | Claude 风格稳健、句式自然；GLM 在中文法律术语上准确但偶有 AI 句式 |
| 案例框 / 场景描述 | DeepSeek | GLM | DeepSeek 倾向具体细节，案例画面感强；注意防过度口语化 |
| 操作步骤 / 清单 | GLM | Claude | GLM 步骤清晰、术语贴；Claude 偶尔过度展开 |
| 导语 / 章首 | Claude 旗舰 | — | 导语需要自然入口感，Claude 旗舰最稳；其他模型容易用"表面上看""更关键的是"开头 |

> 这张表是**起点不是结论**。每跑完一轮横评，根据实际结果更新。

## 输出格式

每次横评完成，给用户一份结构化报告（直接口头 / 写入对话即可，不强求落盘）：

```markdown
## 横评结果：<章节 / 段落标识>

**写作任务**：<一段 / 一章 / 案例框 等>
**参评模型**：GLM / Claude / Qoder-Claude / DeepSeek
**裁判模型**：<GPT-4 / Claude Opus 等，必须不在候选里>

### 排名
1. **<模型 A>** — 总分 X.X / 5
   - 致命问题：<2-3 条>
   - 独特优势：<1-2 条>
2. **<模型 B>** — 总分 X.X / 5
   ...

### 选优建议
- **定稿模型**：<模型 A>
- **回退策略**：<如果 A 不稳怎么办>
- **一句话结论**：<这类写作任务，哪个模型最稳>

### 已记入选型日志
✅ references/model-selection-log.md 已追加本次记录。
```

## 与既有 skill 的边界

| 场景 | 用本 skill | 用其他 skill |
|------|----------|------------|
| 横向对比多个模型的去 AI 化表现 | ✅ | — |
| 单份草稿去 AI 化 | — | `de-ai-polish` |
| 书稿口吻门禁终检 | — | 按 `docs/BOOK-TONE-REVIEW.md` 手动 / `writing-reviewer` |
| 全方位审稿（术语 / 跨章 / 可读性） | — | `writing-reviewer` |
| 选模型 API / Provider | — | `myagents-cli` |
