# 迭代上下文

本文件记录 `legal-case-analysis` 的长期设计上下文，供后续持续迭代时快速理解边界、资源分层和优先方向。

## 一、定位

`legal-case-analysis` 是通用法律分析 Skill，用于把案件材料、咨询材料、证据材料和法律检索结果转化为可复用的法律分析结果、风险评估、诉讼策略、客户说明或结构化法律分析报告。

本 Skill 的核心定位是法律分析引擎。报告只是其中一种交付形态；在服务方案、客户沟通、诉讼策略、合同风险、结案复盘等场景中，法律分析过程都可以作为前置步骤存在。

本 Skill 不绑定具体案由。具体领域规则通过案件材料、法律检索结果和用户目标动态补足。

## 二、核心问题

本 Skill 解决的是律师办案中的通用分析流程问题：

1. 材料如何转化为事实和证据台账。
2. 事实如何映射到请求权基础和构成要件。
3. 争议焦点如何转化为法律检索任务。
4. 检索结果如何回填为可适用的法律规则。
5. 法律规则如何结合事实、证据和抗辩形成结论。
6. 风险、补证和程序路径如何转化为可执行建议。
7. 分析结论如何作为其他 Skill 的输入，而不是被误认为必须生成最终报告。

## 三、分层原则

### SKILL.md

只保留：

- 触发场景和边界。
- 核心原则。
- 输入输出说明。
- `references/` 和 `templates/` 的导航。

不在 `SKILL.md` 中展开完整分析流程、报告章节或模板内容。

### references/

放分析过程和工作方法：

- `workflow.md`：完整工作流。
- `analysis-engine-boundary.md`：分析引擎定位和跨 Skill 协作边界。
- `report-depth-control.md`：按案件复杂度控制输出详略。
- `element-litigation-nine-step.md`：要件式诉讼分析九步法。
- `fact-evidence-extraction.md`：事实与证据抽取。
- `civil-commercial-analysis-workflow.md`：复杂民商事案件分析步骤。
- `issue-spotting.md`：法律问题识别。
- `legal-research-bridge.md`：法律检索衔接。
- `argument-risk-assessment.md`：论证与风险评估。
- `iteration-context.md`：长期迭代上下文。

### templates/

只放最终交付或中间交付可直接套用的格式：

- `legal-analysis-report.md`：法律分析报告。
- `legal-research-query-list.md`：法律检索任务清单。
- `simple-consultation-answer.md`：简版咨询答复。
- `client-summary.md`：客户说明。

## 四、非目标

本 Skill 不做以下事情：

- 不替代 OCR 或文档转换工具。
- 不替代法律检索工具。
- 不直接生成 Word 排版文件。
- 不替代 `legal-proposal-generator` 生成诉讼服务方案、非诉服务方案、法律服务建议书等服务文档。
- 不基于无材料支撑的抽象法律问题输出确定结论。
- 不把单一案由的专项规则写入核心适用边界。

## 五、法律检索原则

涉及具体法条、司法解释、类案、时间效力、时效、管辖、举证责任、执行异议、优先权、损失计算等问题时，应先生成法律检索任务，再基于检索结果分析。

法律检索结果应整理为规则表，说明：

- 来源。
- 规则摘要。
- 适用条件。
- 本案对应事实。
- 支持方向。
- 风险提示。

禁止只堆法条或只凭模型记忆写确定法律结论。

## 六、输出风格

法律分析报告应做到：

- 事实、证据、规则和结论可追溯。
- 有利因素和不利因素并列呈现。
- 风险使用“高/中/低 + 理由”表达。
- 对未提及或待核实事项明确标注。
- 对程序路径和补证动作给出可执行建议。

客户说明应做到：

- 表达通俗。
- 结论靠前。
- 风险明确。
- 行动建议具体。

## 七、后续迭代方向

优先迭代：

1. 脱敏输出模式。
2. 已有检索结果回填规则。
3. 时间效力检查规则。
4. 金额计算和重复计算风险模板。
5. 分析过程工作簿模板。
6. 要件式诉讼分析在常见案由中的校准样例。
7. 不同案件阶段的简化提示词示例。
8. 常见法律关系的专项参考文件。
9. 报告详略控制在真实案件中的校准样例。

## 八、维护规则

- 新增分析方法时，优先放入 `references/`。
- 新增最终交付格式时，放入 `templates/`。
- 新增脚本前先确认是否存在重复、稳定、机械化需求。
- 重要取舍写入 `DECISIONS.md`。
- 用户可见变化写入 `CHANGELOG.md`。
- 后续任务写入 `TASKS.md`。
