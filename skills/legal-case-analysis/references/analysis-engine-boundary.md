# 法律分析引擎定位与协作边界

本文件用于区分 `legal-case-analysis` 与其他法律文档类 Skill，尤其是 `legal-proposal-generator`。

## 一、核心定位

`legal-case-analysis` 解决的问题是：法律上怎么看。

它负责：

- 从材料中提取事实和证据。
- 识别法律关系和争议焦点。
- 拆解请求权基础、构成要件、抗辩和程序问题。
- 生成法律检索任务并吸收检索结果。
- 形成风险判断、证据缺口、策略方向和行动建议。

它不要求每次都生成正式报告。分析结果可以是简短答复、分析底稿、要件表、检索清单、风险清单或完整法律分析报告。

## 二、与 legal-proposal-generator 的边界

`legal-proposal-generator` 解决的问题是：如何把已有材料和分析包装成特定法律服务文档。

| 场景 | 优先使用 |
|------|----------|
| 判断案子能不能打、风险在哪里、需要查哪些法 | `legal-case-analysis` |
| 生成请求权基础、构成要件、证据缺口、抗辩风险 | `legal-case-analysis` |
| 生成简版法律咨询答复 | `legal-case-analysis` |
| 生成诉讼服务方案、非诉服务方案、法律服务建议书 | `legal-proposal-generator` |
| 生成客户沟通报告、结案汇报、内部沟通报告 | `legal-proposal-generator` |
| 已有分析结论，需要改写成对外服务文档 | 先 `legal-case-analysis`，再 `legal-proposal-generator` |

## 三、推荐协作流程

### 1. 先分析再成文

复杂案件或需要专业判断的服务文档，应先完成法律分析：

1. 用 `legal-case-analysis` 输出事实台账、法律问题树、检索清单、要件分析和风险建议。
2. 必要时调用 `yuandian-law-search` 完成法条和类案检索。
3. 将分析结果交给 `legal-proposal-generator`，生成服务方案、建议书、沟通报告或结案汇报。

### 2. 简单文档可直接成文

如果用户只要求把既有沟通记录整理成固定模板，且不需要新增法律判断，可以直接使用 `legal-proposal-generator`。

### 3. 发现法律判断不足时回到分析

文档生成过程中，如发现以下情况，应回到 `legal-case-analysis`：

- 事实、证据和结论无法对应。
- 需要判断胜诉风险、责任成立、请求范围或程序路径。
- 需要引用具体法条、司法解释或类案。
- 客户方案中包含重要法律判断但缺少检索依据。

## 四、输出给其他 Skill 的分析底稿

当分析结果要交给其他 Skill 使用时，优先输出以下结构：

```markdown
# 法律分析底稿

## 一、案件画像
## 二、关键事实和证据
## 三、法律问题树
## 四、请求权基础或分析路径
## 五、要件-事实-证据对应表
## 六、法律检索结果摘要
## 七、主要风险和反向观点
## 八、建议采用的文档口径
## 九、待补充材料
```

这类底稿不追求正式报告表达，重点是让后续文档生成有可靠分析依据。
