# 通用法律分析工作流

本文件定义 `legal-case-analysis` 的标准处理流程。遇到复杂案件时，按本流程分阶段推进；遇到简单咨询时，可合并相邻步骤，但不得跳过事实核验和法律依据校验。

本技能是法律分析引擎，不要求每次都生成正式报告。交付物可以是分析底稿、检索清单、咨询答复、客户说明，也可以作为 `legal-proposal-generator` 等文档生成 Skill 的前置输入。

## 一、启动阶段

开始分析前，先按 `report-depth-control.md` 判断案件复杂度和报告深度。简单咨询、标准案件和复杂案件的处理深度不同，不应一律套用同一份长报告。

如果任务涉及诉讼、仲裁、执行异议、上诉/再审或其他争议解决，先判断案件类型再选分析支架：

- **民事/民商事请求权争议**：读取 `element-litigation-nine-step.md`，用要件式九步法校准请求权基础、构成要件、事实、证据、证明责任和抗辩。
- **刑事案件**：读取 `criminal-case-analysis.md`，用刑事分析八步法（审查起诉骨架 + 辩护对抗检验）校准犯罪构成（默认四要件）、罪数与形态、证据（确实充分、排除合理怀疑）、量刑与程序合法性；违法/责任阻却事由用三阶层（该当性→违法性→有责性）在分析内逐层审查。
- **行政案件**：以九步法为基础，按行政法律关系调整（合法性审查、职权/程序/证据），必要时生成检索任务补足。

### 1. 识别用户目标

先判断用户真正要什么：

| 目标 | 输出侧重 |
|------|----------|
| 是否能起诉 | 请求权基础、证据缺口、胜诉可能性 |
| 如何应诉 | 对方主张拆解、抗辩路径、举证计划 |
| 是否上诉/再审 | 裁判错误、程序瑕疵、新证据、改判空间 |
| 如何谈判 | 筹码、底线、风险、替代方案 |
| 给客户汇报 | 结论清楚、风险通俗、行动建议明确 |

### 2. 识别案件阶段

程序阶段会影响分析重点：

| 阶段 | 重点 |
|------|------|
| 咨询阶段 | 材料完整性、初步权利基础、补证方向 |
| 诉前阶段 | 诉请设计、管辖、保全、催告、时效 |
| 一审阶段 | 争点组织、证据提交、举证责任 |
| 二审阶段 | 一审错误、上诉理由、新证据限制 |
| 执行阶段 | 执行异议、财产线索、参与分配、优先权 |
| 非诉阶段 | 风险分配、谈判策略、文件修改 |

## 二、材料处理阶段

### 1. 建立材料清单

逐项记录：

- 文件名称。
- 文件类型。
- 来源。
- 日期。
- 涉及主体。
- 证明目的。
- 可靠性初评。
- 是否需要原件或进一步核实。

### 2. 建立事实台账

将材料转化为事实，而不是直接摘要。每条事实至少包含：

- 时间。
- 主体。
- 行为或事件。
- 来源材料。
- 证据强度。
- 是否存在争议。

### 3. 标注缺失信息

对未提及但影响判断的信息，标注“待补充”。不要替用户补写成确定事实。

## 三、问题识别阶段

围绕用户目标拆解法律问题：

1. 我方想获得什么结果。
2. 该结果需要满足哪些法律条件。
3. 现有事实能满足哪些条件。
4. 哪些条件证据不足。
5. 对方最可能攻击哪些环节。
6. 是否存在替代路径。

诉讼或争议解决场景中，问题识别应进一步转化为要件表：

- 请求或抗辩依据。
- 构成要件。
- 对应事实。
- 支撑证据。
- 证明责任。
- 对方抗辩。
- 风险和补强动作。

## 四、法律检索阶段

先形成检索任务，再调用法律检索工具。检索任务应包括：

- 问题背景。
- 关键词。
- 需要检索的法条。
- 需要检索的案例类型。
- 时间效力要求。
- 希望验证的初步判断。

检索后将结果整理成规则表，再进入论证。

## 五、论证阶段

每个争点按同一结构处理：

1. 结论。
2. 事实基础。
3. 证据基础。
4. 法律依据。
5. 适用分析。
6. 反向观点。
7. 风险等级。
8. 行动建议。

## 六、交付阶段

根据用途选择报告形态：

| 用途 | 模板 |
|------|------|
| 仅供后续文档生成或内部讨论 | 可输出分析底稿，不必套正式报告模板 |
| 律师内部研判 | `templates/legal-analysis-report.md` |
| 客户汇报 | `templates/client-summary.md` |
| 先检索再分析 | `templates/legal-research-query-list.md` |

交付前必须说明：

- 哪些结论已有材料支持。
- 哪些结论依赖法律检索。
- 哪些事项仍需补充材料。
- 哪些风险无法消除。

## 七、交付前检查

- 是否明确委托人角色、目标用途和程序阶段。
- 是否区分证据、陈述、争议材料和待补充事实。
- 是否形成时间轴或关键事实表。
- 是否列明法律问题树。
- 诉讼或争议解决场景是否完成要件式分析。
- 是否生成或吸收法律检索结果。
- 是否避免引用过时法律规则。
- 是否逐项分析对方抗辩。
- 是否给出风险等级和补强证据建议。
- 是否把未提及事项标注为“未提及/待补充”。
- 是否适合转为 Word 正式报告。
