# 技术决策记录

本文档记录 `article2book` 技能设计过程中的关键决策与工作日志。

## [DEC-001] - 2026-04-02 - 输出重点放在“成书判断 + 结构方案”，而不是文章摘要

### 背景

用户的真实需求不是简单汇总文章，而是判断一个既有文章目录能否被提炼为可出版的书籍项目，并给出结构化整合方案。

### 选项

1. **做成文章总结器**
   - 优点：实现简单
   - 缺点：无法回答“能否成书”和“怎么重组”的关键问题

2. **做成书稿策划器**
   - 优点：直接回应出版判断、目录结构、读者定位和删改建议
   - 缺点：需要更强的评估标准与输出模板

### 决策

选择 **选项 2：做成书稿策划器**。

### 理由

1. 用户已经有相当多现成内容，真正缺的是“收束”和“结构化”
2. 成书不是摘要工作，而是主题判断、读者定位和章节重组工作
3. 这样设计更符合文章资产再利用的真实场景

### 影响

- 标准输出改为书稿策划包，而不是摘要合集
- 增加成书可行性评估和目录映射两个核心文件

## [DEC-002] - 2026-04-02 - 采用“源目录优先，archive 回退”的输出策略

### 背景

用户希望技能对任意文章目录都可复用。如果把输出路径写死到某个固定目录，后续跨设备、跨目录使用会非常别扭。

### 选项

1. **固定输出到某个个人路径**
   - 优点：单机使用方便
   - 缺点：不可移植

2. **要求用户每次都手动指定输出路径**
   - 优点：灵活
   - 缺点：交互成本高

3. **源目录优先，archive 回退**
   - 优点：成书策划结果与原文章目录相邻，同时兼容不可写目录
   - 缺点：需要清楚说明双路径规则

### 决策

选择 **选项 3：源目录优先，archive 回退**。

### 理由

1. 书稿整理结果最好贴着原文章资产存放，方便反复修订
2. 回退策略能兼容只读目录和统一归档需求
3. 该方式比硬编码个人路径更利于复用

### 影响

- `SKILL.md` 中明确输出目录规则
- 新增 `archive/.gitkeep`

## [DEC-003] - 2026-04-02 - 增加标准库扫描脚本，先建立素材清单再做判断

### 背景

如果直接阅读大量素材全文再人工整理，成本高且容易漏项。先建立素材清单，有助于识别主题分布、规模和边缘内容。

### 选项

1. **只靠人工读取**
   - 优点：灵活
   - 缺点：面对几十篇文章时容易漏读、漏记

2. **增加轻量扫描脚本**
   - 优点：先把候选文章、标题、预览、规模统一索引出来
   - 缺点：需要维护一个小脚本

### 决策

选择 **选项 2：增加轻量扫描脚本**。

### 理由

1. 先盘点素材资产，再做主题判断，更符合内容策划流程
2. 标准库脚本没有额外依赖，适合本地快速运行
3. 结构化索引可作为后续章节映射和删改判断的基础

### 影响

- 新增 `scripts/build_article_inventory.py`
- 标准输出中加入 `processed/article_inventory.csv/jsonl`

## [DEC-004] - 2026-04-02 - 技能以“现有素材重组”为主，不以“从零写书”为主

### 背景

用户进一步明确，这个技能的目标并不是从零生成一本书，而是基于已经存在的文章、视频教程、课程讲稿等素材，把内容重组为书籍形式。

### 选项

1. **保留“文章整理成书”的窄定义**
   - 优点：边界更简单
   - 缺点：无法覆盖视频教程、讲稿、逐字稿等真实来源

2. **扩展为“现有内容素材结构化成书”**
   - 优点：更贴近作者已有内容资产的真实形态
   - 缺点：需要明确非文本素材需先转写

### 决策

选择 **选项 2：扩展为“现有内容素材结构化成书”**。

### 理由

1. 很多可出版内容的起点并不是文章，而是课程、视频、分享或逐字稿
2. 真正的价值是“把已有表达重新编排为书籍结构”，不是“凭空产书”
3. 这样定义后，技能更适合作为内容资产再开发工具

### 影响

- `SKILL.md` 明确加入课程讲稿、视频教程逐字稿、演讲提纲等典型输入
- 标准输出改称“候选素材总表”
- 扫描脚本增加 `.srt`、`.vtt` 支持，并新增 `source_type` 字段

## [DEC-006] - 2026-04-02 - Agent 通读优先，脚本仅作辅助索引

### 背景

用户明确指出，成书判断最重要的是 Agent 对全文内容的阅读理解，而不是依赖脚本先扫目录。脚本可以做索引，但不应该主导主题判断。

### 选项

1. **保留“脚本先扫，再人工抽读”的流程**
   - 优点：更快
   - 缺点：容易把主题判断建立在不充分阅读上

2. **改为“Agent 通读优先，脚本仅作辅助”**
   - 优点：更贴近真正的阅读理解过程
   - 缺点：耗时更高，需要明确批次阅读方法

### 决策

选择 **选项 2：Agent 通读优先，脚本仅作辅助**。

### 理由

1. 成书判断依赖的是母题、论证、重复度和章节潜力，这些都需要全文理解
2. 文件名、摘要和统计字段只能帮助查漏，不能替代阅读
3. 对课程讲稿和逐字稿这类口语素材，尤其需要 Agent 自己判断其可书稿化程度

### 影响

- `SKILL.md` 的主流程改为先通读全部候选文本素材
- 新增 `references/agent-reading-protocol.md`
- 输出模板增加 `processed/agent_reading_notes.md`
- `scripts/build_article_inventory.py` 降级为可选辅助工具

## [DEC-007] - 2026-04-02 - 通读后先做低质量内容排除

### 背景

用户指出，在 Agent 通读完素材后，还应先筛掉一些老旧、浅层、低质量或明显较弱的内容，否则后续目录设计会被噪音拖累。

### 选项

1. **把全部素材都纳入后续聚类**
   - 优点：不容易漏材料
   - 缺点：低质量旧内容会稀释主线，降低成书质量

2. **通读后先做“保留 / 降权 / 排除”筛选**
   - 优点：先做减法，再做结构化，更接近真实编辑流程
   - 缺点：需要明确筛选标准，避免误删有价值旧文

### 决策

选择 **选项 2：通读后先做“保留 / 降权 / 排除”筛选**。

### 理由

1. 一本书不需要收进所有旧内容，而是要留下最能支撑母题的内容
2. 老内容不是问题，老而浅、过时失效、重复且更弱才是问题
3. 先筛掉噪音，后续主题聚类和目录设计会更清晰

### 影响

- `SKILL.md` 新增“通读后先筛选”的明确步骤
- 新增 `references/content-screening-rubric.md`
- 输出模板增加 `00A-排除与降权清单.md`

## [DEC-008] - 2026-04-02 - 默认输出改为单一“书稿策划意见”

### 背景

在真实目录上跑完第一版后，发现虽然多文件分析包足够完整，但用户在首次判断阶段并不需要同时阅读这么多文件。过多拆分文件会放大认知负担，反而削弱“先确认方向”的效率。

### 选项

1. **继续默认输出多文件分析包**
   - 优点：结构完整，可追溯性强
   - 缺点：文件过多，首次阅读成本高

2. **默认只输出一份主意见，支撑文件按需生成**
   - 优点：更贴近真实决策路径，先看结论，再决定是否展开细节
   - 缺点：需要在技能中额外说明哪些文件属于可选支撑

### 决策

选择 **选项 2：默认只输出一份 `书稿策划意见.md`，支撑文件按需生成**。

### 理由

1. 用户在第一阶段真正需要的是一份可判断、可修改、可确认的主意见
2. 素材总表、筛选清单和阅读笔记更适合作为后备依据，而不是默认阅读入口
3. 这样更利于把 skill 接到第二阶段的直接写书动作上

### 影响

- `SKILL.md` 的标准输出调整为“单文件轻交付”
- `references/output-template.md` 改为以 `书稿策划意见.md` 为默认模板
- `processed/agent_reading_notes.md`、素材总表和筛选清单降级为可选支撑文件

## [DEC-009] - 2026-04-02 - 引入“先策划，后直出全书”的两阶段流程

### 背景

用户明确希望先拿到一份整体意见，确认方向后不要再经历冗长中间步骤，而是直接进入全书重构，尽量一步到位地得到完整书稿。

### 选项

1. **技能只负责策划，不负责后续书稿重构**
   - 优点：边界单纯
   - 缺点：用户还要再发起第二套流程，衔接成本高

2. **技能内置两阶段流程：先策划，确认后直接出全书初稿**
   - 优点：更贴近真实使用路径，也更利于复用通读结果
   - 缺点：需要说明阶段切换条件和统稿责任

### 决策

选择 **选项 2：技能内置两阶段流程**。

### 理由

1. 用户需求不是“分析完就停”，而是“分析后立刻可进入重构”
2. 通读与筛选的结果若不继续用于写书，很多中间理解会被浪费
3. 两阶段放在同一 skill 中，更容易统一母题、目录、术语和文风

### 影响

- `SKILL.md` 新增第二阶段：确认后输出 `全书初稿.md`
- `references/output-template.md` 新增 `全书初稿.md` 模板说明
- 允许用 subagent 分章并行起草，但由主 Agent 负责最终统稿

## [DEC-010] - 2026-04-02 - 默认不外显内部分析痕迹

### 背景

在第一版真实交付中，虽然已经把默认输出压缩为单份意见，但筛选细表、阅读笔记和过程性文件仍然容易被继续显性输出，用户首次判断时仍会承受不必要的阅读负担。

### 选项

1. **继续保留过程性文件作为常规可见输出**
   - 优点：可追溯性强
   - 缺点：容易把内部工作痕迹误变成用户默认阅读入口

2. **把过程性文件进一步内化，只在必要时显性输出**
   - 优点：更符合“先看意见、再决定是否展开”的真实使用路径
   - 缺点：需要在主意见中压缩保留足够的筛选依据

### 决策

选择 **选项 2：把过程性文件进一步内化**。

### 理由

1. 用户真正要先判断的是方向，不是阅读完整分析过程
2. 筛选原则和删改动作可以压缩进主意见，不必默认拆成单独文件
3. 这样更利于从第一阶段无缝衔接到第二阶段的整书重构

### 影响

- `SKILL.md` 新增“内部痕迹不默认外显”的判断原则
- `references/output-template.md` 要求把“保留 / 合并 / 排除原则”默认写入 `书稿策划意见.md`
- `附录-排除与降权清单.md` 明确为非默认产物

## [DEC-012] - 2026-04-03 - 示例与验证记录不保留用户本地绝对路径

### 背景

用户指出，skill 中的示例不应暴露其本地目录路径。该问题不仅出现在 `SKILL.md` 的示例请求里，也会连带影响变更记录中的真实验证表述。

### 选项

1. **保留真实绝对路径**
   - 优点：记录精确
   - 缺点：会把用户本地目录信息固化在技能文档中

选择 **选项 2：改为通用占位或泛化表述**。

### 理由

1. 示例的任务是传达用法，不需要绑定某个真实机器路径
2. 技能级文档应尽量可复用，而不是带着单个用户环境痕迹
3. 验证事实可以保留，但不需要暴露验证时使用的具体本地目录

## [DEC-013] - 2026-04-03 - 取消 `archive` 回退，永远默认输出到源目录

### 背景

用户明确要求 `article2book` 不再使用技能目录内的 `archive/` 作为回退输出位置，而是始终把结果写回素材所在目录。

1. **保留“源目录优先，archive 回退”**
   - 优点：兼容不可写目录和集中存放

## [DEC-014] - 2026-05-17 - 从成书策划升级为内容资产再组织

### 背景

用户希望 `article2book` 写得更有泛化性。现有版本已经能处理文章、讲稿和逐字稿的成书策划，但默认判断仍以“能不能成一本书”为中心。真实内容资产整理场景中，很多素材更适合转化为课程、小册子、系列文章、实务手册或知识库。

### 选项

1. **继续强化成书路径**
   - 优点：边界清晰，延续既有定位
   - 缺点：容易把不适合成书的素材也强行导向书稿

2. **升级为内容资产再组织，保留成书作为默认主路径**
   - 优点：既保留已有能力，又能覆盖课程、手册、知识库等更自然的转化结果
   - 缺点：需要新增形态判断、素材类型处理和输出模板字段

### 决策

选择 **选项 2：升级为内容资产再组织，保留成书作为默认主路径**。

### 理由

1. 用户的核心需求是把已有内容资产变成更有价值的结构化成果，不一定总是一本书
2. 先判断最佳内容形态，可以减少错误成书、硬拼目录和过度写作
3. 保留 `article2book` 名称与两阶段流程，能兼容既有触发习惯和历史文档

### 影响

- `SKILL.md` 的定位扩展为“内容资产再组织”
- 新增内容形态判断树和快速盘点 / 标准策划 / 深度重构三档模式
- 新增 `references/content-productization-models.md` 和 `references/source-type-handling.md`
- `references/output-template.md` 增加最佳内容形态、不建议形态、转化路径和下一步产物
