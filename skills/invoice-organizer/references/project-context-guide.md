# 项目上下文回溯指南

报销清单"报销信息"表的事由/案号/日期/路线等，优先从发票所属项目的上下文提取，而非留空或依赖用户口述。

## 项目根识别

从发票所在路径向上逐级回溯，命中下列任一标志文件即视为项目根：

- `项目信息.md`（诉讼/法律项目主信息文件，new-case 体系）
- `待办事项.md`
- `README.md`（部分项目以 README 作说明）

回溯一般 2～4 级（发票常归档于 `项目/NN - 其他材料/报销/` 深层）。到工作文档根（含 `003 - 诉讼仲裁` 等业务编号目录的父级）仍未命中，视为非项目目录，事由向用户确认。

## 字段提取映射

| 清单"报销信息"字段 | 项目上下文来源 |
|---|---|
| 事由 | 项目/案件名称 + 触发本次出行的事件 |
| 案号 | 项目信息.md"法院案号" |
| 日期 | 与发票业务日期匹配的项目时间线事件 |
| 路线 | 事件地点（如法院所在地）↔ 常驻地 |
| 报销单位（发票抬头） | 委托人/客户（与发票购买方交叉验证） |
| 出差人 | 承办律师/项目出差人员 |

## 日期匹配推断事由

把每张发票的**业务日期**（乘车日/住宿日，非开票日期）与项目时间线比对，找时间重合的事件作为事由：

- 发票 2026-06-15～17 + 项目"2026-06-16 第一次开庭（石家庄）" → 事由"赴石家庄参加第一次开庭"
- 发票日期无对应事件 → 事由标"待确认"，向用户核实

## 各类项目的信息文件

- 诉讼案件（`003 - 诉讼仲裁/`）：`项目信息.md`（案号、当事人、时间线、承办律师、费用跟踪）
- 潜在项目/咨询（`011 - 潜在项目/`）：项目目录下说明/服务方案文件
- 法律项目（`004 - 法律项目/`）：同上
- 非项目目录（如纯个人消费）：无项目根标志，事由由用户直接给出

## 边界与安全

- 仅读取、不修改项目信息文件
- 项目信息中的当事人身份证号、电话等敏感信息不写入清单（清单只留姓名）
- 上下文缺失的字段标注"待确认"，不臆造事由或金额
