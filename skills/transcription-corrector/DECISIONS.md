# Transcription Corrector 决策记录

## 决策记录

### [DEC-001] - 2026-06-07 - 不输出课程大纲，与下游课程整理类工作流严格分工

**背景**
下游课程整理类工作流已经具备"用户词典纠错 + 整理转录稿 + 生成课程"的能力；本 skill 设想中也包含"读转录稿 → 校对"这一环。

**决策**
本 skill 只做"raw → 已校对可读稿"这一段预处理：
- 纠错（人名、产品名、术语、数字）
- 轻度润色（发言人合并、标点、段落切分）
- **不**生成课程大纲
- **不**重组章节
- **不**改写为书面课程
- **不**重写为总结

如需更深度的"转录稿 → 课程"加工，下游进一步加工。

**理由**
单一职责更易维护；与下游的边界清晰；高频独立场景（只想要校对稿，不要课程）不再被强制打包到课程整理流程里。

**影响**
SKILL.md 明确"不适用场景"列出重写课程、总结等，并把课程整理类下游环节列为接力节点。不同流程共享用户词典 YAML 格式，但工作流独立。

### [DEC-002] - 2026-06-07 - 词典是"目标值+权重"，不是触发器

**背景**
用户指出：ASR 错误无穷无尽（大写、空格、漏字、错位、同音字、形近字、断词、音节吞并…），任何静态"错误词 → 正确词"映射表必然漏判。词典条目登记的应是"目标值"——AI 在通读全文时基于上下文判断是否替换为该目标值。

**决策**
- 词典项是"正确写法"，告诉 AI 目标值是什么
- 词典项**不直接触发替换**：AI 仍须先做上下文判断
- 词典项**提高候选替换的胜出权重**：当多个候选写法竞争时，登记项优先
- 三重确认才替换：
  1. 词典里有对应目标值
  2. 上下文明确指向该目标值所代表的对象
  3. 形式漂移证据成立（大小写/空格/连字符/漏字/同音/形近/断词）

**理由**
避免把 AI 应做的判断固化成静态映射；与本 skill 的设计原则保持一致；处理无穷无尽的 ASR 错误变体。

**影响**
SKILL.md Step 0 明确"词典的角色"；`references/correction_patterns.md` 从原计划的"映射表"改写为"判断准则"；词典 YAML 注释中给出三重确认清单。

### [DEC-003] - 2026-06-07 - 原始文件保持不动，所有产物自包含在 archive/

**背景**
纠错是辅助动作；任何"误改"风险都应优先于"流程效率"。

**决策**
- 原始转录文件**绝不被修改**
- 所有纠错/润色结果输出到 `archive/YYYYMMDD_HHMMSS_{原文件 basename}/`
- 每次运行产生独立时间戳目录，避免覆盖历史
- `correction_log.md` 记录每条决策（含"未处理"区），保证可追溯
- `meta.json` 记录处理时间、词典来源、润色开关

**理由**
任何"覆盖原文件"的设计都有误改风险；校对场景下"保留原始证据"比"流程简洁"更重要；时间戳目录让用户可以横向对比多次纠错效果。

**影响**
SKILL.md Step 5 明确输出位置规则；archive/ 目录加入 `.gitignore`-friendly 模式（与其他上游转录工具的归档风格一致）。

### [DEC-004] - 2026-06-07 - 轻度润色默认关闭，避免越界改写

**背景**
"纠错"和"润色"边界模糊——口语化清理、口头禅去除、段落合并都是"看起来无害"的操作，但累积起来会改变原文语气与说话人风格。

**决策**
- 润色模式是可选开关，**默认关闭**
- 开启后只做三类最小动作：发言人合并、标点规范、极长段切分
- **不做**：改写为书面语、总结、补全、删减"啰嗦"内容
- 每个动作都需保留原话、不改事实

**理由**
纠错与润色是不同性质的工作：纠错是"消除错误"，润色是"风格调整"；用户场景里前者 100% 需要，后者 0~30% 需要，且必须明确开关。

**影响**
SKILL.md 工作流拆为两种模式：纠错模式（默认）、纠错+轻度润色（用户显式开启）。

### [DEC-005] - 2026-06-07 - 配置文件解耦：example 入仓、真实 git ignore

**背景**
skill 准备对外发布。需要区分"通用模板"和"个人配置"：
- 通用术语（WorkBuddy、Claude Code 等行业公认工具）所有用户都能用 → 应入仓
- 个人项（作者本人姓名、律所名、客户代号）仅本人需要 → 不应入仓

**决策**
- `config/user_dictionary.example.yaml` 只放通用术语，作为公开模板
- 本地用户复制 example → 改名为 `user_dictionary.yaml` → 追加个人项
- `config/.gitignore` 显式声明 `user_dictionary.yaml` 不入仓（与根 .gitignore 的 `**/config/*.yaml` + `!**/config/*.example.yaml` 配合）
- SKILL.md 顶部加"首次使用"小节
- SKILL.md 加"配置解耦原则"小节，给出"换另一个用户 clone，他需要这一项吗？"的判定标准

**理由**
防止个人项通过公开仓库泄露；同时让任何 clone 仓库的用户拿到一个开箱即用的最小词典。

**影响**
example.yaml 默认内容只保留通用工具/术语；个人项全部走本地化。

**背景**
面向公开发布的 SKILL.md / CHANGELOG.md / TASKS.md / DECISIONS.md / references/ / example.yaml 等文件，正文与描述示例中不应出现作者本人姓名。但 SKILL.md frontmatter 的 `author` 字段、AGENTS.md 规定的 LICENSE.txt 版权信息除外。

**决策**
- SKILL.md frontmatter 的 `author` 字段保留
- SKILL.md / CHANGELOG.md / TASKS.md / references/ / example.yaml 正文中"作者本人姓名"作为描述示例的提及全部移除，改用通用占位

- DECISIONS.md 工作日志里不写具体 AI 名，只保留日期

**理由**
开源项目对作者身份的暴露应限于元数据（frontmatter / LICENSE），正文示例不应再带个人姓名；让"换另一个律师 clone 仓库"的场景下示例仍合理。

**影响**
所有公开文件的"示例/描述"段已清理，仅剩 frontmatter 与 LICENSE 含作者标识。

### [DEC-007] - 2026-06-07 - TXT 输入支持，Step 0.5 做格式转换

**背景**
实际工作流中 raw 转录稿常以 `.txt` 形式给出（本地转录引擎、云端 ASR、人工导出等），而非 Markdown。skill 默认假设输入是 Markdown，需要扩展。

**决策**
新增 Step 0.5：当输入是 `.txt` 时，做以下转换后再进入 Step 1
- 识别发言人 + 时间戳行（正则：`^(发言人|说话人|Speaker|S)\s*[\d一二三四五六七八九十]+[\.、]?\s*(\d{1,2}:\d{2}(?::\d{2})?)$`）→ 转 Markdown 加粗
- 段落间补空行
- 不删除任何文字
- 不主动升级为标题层级、不识别列表
- 输出 `{原文件}_converted.md` 中间产物到 archive 子目录
- 原始 `.txt` 保持不动

**理由**
raw txt → markdown 包装是"格式转换"，属于本 skill 边界内（与"纠错"同属"读起来对"环节）。保守策略：只做格式包装，不识别内容；让下游处理环节去做深度处理。

**影响**
SKILL.md 新增 Step 0.5 完整说明（含示例）；CHANGELOG.md 加 v1.0.1；TASKS.md 加"用 .txt 输入试跑"待办。

### [DEC-008] - 2026-06-08 - 基础空白清理作为独立步骤（Step 3.5），默认开启

**背景**
实战中 raw 转录稿常出现"行首行尾多余空格"、"连续空行"、"专有名称内部断裂空格"、"中文数字+空格+级/章/节"（如"二 级"）等格式噪音。这些空白噪音不影响语义，但阻碍阅读。问题是：把这类清理放在 Step 3（词典替换）里会污染"高置信替换"的语义边界；放在 Step 4（润色）里又会让"无损"操作与"风格调整"操作混淆。

**决策**
新增 **Step 3.5：基础空白清理** 作为独立步骤，**默认开启**（因为是无损的格式标准化），与 Step 3（信息修正）、Step 4（风格调整）严格区分：

| 步骤 | 性质 | 默认开关 | 改语义？ | 改风格？ |
|---|---|---|---|---|
| Step 3 词典替换 | 信息修正 | 开启 | ✓ | ✗ |
| **Step 3.5 基础空白清理** | **格式标准化** | **开启（无损）** | **✗** | **✗** |
| Step 4 发言人合并 | 风格调整 | 关闭 | ✗ | ✓ |
| Step 4 标点规范 | 风格调整 | 关闭 | ✗ | ✓ |
| Step 4 段落切分 | 风格调整 | 关闭 | ✗ | ✓ |

**白名单动作**（仅做这些）：
- 行首/行尾多余空格删除
- 连续空行压缩为单个空行
- 英文单词间全角空格 → 半角空格
- 半角空格夹在中文之间 → 删除
- 中文数字 + 空格 + 级/章/节 → 合并（如"二 级" → "二级"）
- 中文标点前后多余空格 → 删除
- 英文标点后多余空格（≥2 个）→ 压缩为 1 个

**专有名称内部断裂空格**的合并（如"Work Buddy" → "WorkBuddy"）**仍走 Step 3 词典替换路径**，不归 Step 3.5——因为这需要上下文判断 + 词典对齐，超出"纯格式"的范畴。

**理由**
- "格式标准化" 与 "信息修正" 是不同性质的工作：前者无损、后者可能有信息损失；混在一起会让校对日志（correction_log.md）失去"高置信"信号的清晰度
- 把 Step 3.5 设为默认开启，是因为这些动作对原文 100% 安全（无信息损失），用户不需要为此做开关决策
- 严格白名单 + "严格不做"小节，避免与 Step 4 润色越界

**影响**
SKILL.md 新增 Step 3.5 完整小节（含清理动作清单、标点周围空白规则、严格不做清单、与 Step 3/4 的边界）；references/correction_patterns.md 新增"§6 基础空白清理规则"小节；correction_log.md 增加"基础空白清理"区与"高置信替换"区分开。

### [DEC-009] - 2026-06-08 - 输出策略扩展为双写：主归档 + 源文件目录镜像

**背景**
原 [DEC-003] 设计为"原始文件保持不动，所有产物自包含在 archive/"。实战中用户经常需要在 Finder / 文件管理器中直接看到"已校对版"，而每次都进 skill 内部 archive/ 翻找时间戳目录不直观。

**决策**
扩展 Step 5 为**双写策略**：

- **5.1 主归档**（必出）：保留原 [DEC-003] 设计，输出到 `{skill 目录}/archive/{YYYYMMDD_HHMMSS}_{原文件 basename}/`，承担"可追溯 + 校对日志 + 元数据"的完整档案职责
- **5.2 源文件目录镜像**（必出）：在原始文件所在目录输出一份 `{原文件 basename}_corrected.md`，作为易访问副本；polished 副本与 correction_log **不镜像**——它们的主归档位置就在 skill 内部

**冲突处理**：
- 源文件目录已存在同名 `_corrected.md` → 用时间戳后缀区分（`_corrected_20260608_153022.md`），避免覆盖历史校对结果
- 源文件目录只读 / 不可写 → 静默跳过源文件目录镜像，仅输出主归档；在 `meta.json` 记录 `source_mirror: "skipped"`
- 源文件路径无法解析（如 stdin 输入）→ 同样静默跳过源文件目录镜像

**原始文件保持不动**——双写策略只生成"新文件"，绝不就地修改原文件。

**理由**
- 主归档 = "完整档案"（log / meta / 多个产物）
- 源文件目录镜像 = "易访问副本"（用户日常文件管理中可见）
- 双写内容字节级一致，但用途不同：前者服务于"可追溯性"，后者服务于"日常可见性"
- 冲突用时间戳后缀 + 静默跳过 + meta 记录，避免破坏性行为

**影响**
SKILL.md 重写 Step 5 为 5.1 + 5.2 双写策略；meta.json schema 新增 `source_mirror` 字段（值：`"ok"` / `"skipped"` / `"conflict_<timestamp>"`）；CHANGELOG 加 v1.0.2 修订记录。

### [DEC-010] - 2026-06-08 - 口语词精简作为独立步骤（Step 3.6），默认开启

**背景**
v1.0.0 时期，references §3 "不替换的边界" 明确不删口语词（"呃/啊/那个/就是说"），SKILL.md Step 4 润色边界表也写"不改语气词、不删'呃/啊'等自然语流标记"。但实战中转录稿口语词密度高（分析反馈：原文 171 处"呃"、57 处"啊"、40 处"嗯"），严重影响阅读体验。v1.0.2 用户反馈"常规口语词汇可以精简或删除"，且明确"默认开启"。

**决策**
新增 **Step 3.6：口语词精简** 作为独立步骤，**默认开启**（用户明确要求）。与 Step 3.5 基础空白清理同级，属于"读起来对"操作。

**白名单**（仅删独立出现且无语义负担的纯填充词）：
- 独立出现的"呃/啊/哦/哎"（认知/感叹/过渡停顿）
- 独立出现的"那个/就是说/然后呢/对吧/你知道吗/怎么说呢/这样子/其实呢/但是呢"（连接/反问/思考/总结/转折填充）

**保留规则**（不删）：
- "嗯/对/是的/不是/好的/是" 作表态回答
- "然后" 作逻辑连接词
- "认知停顿 + 完整内容" 整段保留
- 段首/段尾的关键节奏标记
- 紧邻关键名词/动词的强调用法
- 实义词一律不删

**为什么不属于 Step 4 润色**：
- Step 4 是"风格调整"（发言人合并、标点规范、段落切分）—— 会改说话风格
- Step 3.6 是"语言字符级清理"—— 只删白名单纯填充词，不删实义、不改风格
- 与 Step 3.5 基础空白清理（只动空白字符）性质相同：默认开启、无损

**判断原则**：
- 与 Step 3 一致的"通读 + 三重确认"思路
- 三重确认：① 词类属于白名单；② 当前上下文独立使用、承担实义否；③ 删后不破坏态度/时序/强调
- 任一不满足 → 不删；模糊场景一律保留

**严格不做**：
- 不删整句、不删段落、不改写为书面语
- 不删实义词、不删关键信息（数字/人名/专有名/术语）
- 不删"认知停顿 + 完整内容"的整段
- 不删段首/段尾的关键节奏标记

**理由**
- 实战中口语词密度高（"呃" 171 次 / "啊" 57 次 / "嗯" 40 次），默认删比每次手动开关更省事
- 严格白名单 + 三重确认 + 严守保留规则，能在不破坏原意的前提下显著提升阅读体验
- 校对日志（correction_log.md）"口语词精简"区集中记录，便于事后审计

**影响**
SKILL.md 新增 Step 3.6 完整小节；references 新增 §7 口语词精简规则（5 小节）；correction_log.md schema 扩为 4 区（高置信替换 / 基础空白清理 / 口语词精简 / 未处理）；CHANGELOG 加 v1.0.3 修订记录。

### [DEC-011] - 2026-06-08 - example.yaml 扩展为"常见技术名词 + 常见法律术语"收纳范围

**背景**
v1.0.0 [DEC-005] 设立配置解耦原则时，example.yaml 故意保持极简（只放 WorkBuddy / Claude Code / Codex / skill / Skill 五项），让"换另一个用户 clone 也需要"的判定标准显式触发。v1.0.3 用户反馈"example.yaml 可以把常见技术名词放进去，还有常见法律相关用语"——希望扩大公开模板的收纳范围。

**决策**
example.yaml 扩展为三类收纳：

1. **常见技术名词**（行业公认、ASR 容易听错的产品/平台/格式/缩写）：
   - AI 编码与智能体：Cursor、Anthropic、OpenAI、Coze、Ollama、Manus、Moltbook
   - 国产 / 常用大模型：DeepSeek、腾讯元宝、豆包、智和、Alpha
   - 笔记 / 知识管理：Obsidian、Flomo、Cubox
   - 转录 / 文档处理：FunASR、MinerU
   - 协作 / 通讯：Discord、Slack
   - 工程 / 工具：Docker、Playwright、MonoRepo、clawhub、GitHub、Markdown
   - 常见英文缩写：MCP、AI

2. **常见法律术语**（法律行业通用、ASR 容易听错为同音字/形近字）：
   - 知识产权类：知识产权、专利、商标、著作权
   - 合同/争议解决：合同、诉讼、仲裁
   - 主体/程序：代理、律师、当事人

3. **通用术语**：skill、Skill

**明确不放的**（避免触发过度替换风险）：
- 极通用词：法律、法院、法官、律师、合同、证据、判决、上诉、辩护、司法解释
- 个人工作栈工具：仅本机使用的特定技术栈产品（即使"行业"可能认识，归"个人工作栈"更准确）
- 个人身份：人名、律所名、客户代号

**判定标准（保持）**：
- "换另一个用户 clone 这个仓库，他需要这一项吗？"
- ✅ 需要（行业公认）→ example.yaml
- ❌ 不需要（个人化）→ 本地 user_dictionary.yaml

**理由**
- 用户实战反馈 example.yaml 太精简，新用户 clone 后需要手动追加大量常用术语

### [DEC-014] - 2026-06-08 - Step 编号体系重构为 Phase 分组

**背景**

v1.0.4 的 SKILL.md 用 `Step 0 / 0.5 / 1 / 2 / 3 / 3.5 / 3.6 / 4 / 5` 整数小数混用编号，存在两个问题：
1. `Step 0.5 / 3.5 / 3.6` 小数后缀让人误以为是 Step 3 的子步骤，但实际和 Step 4（润色）同级
2. "改字 / 改空白 / 删字 / 改结构"四种性质不同的操作在序号上没层次区分

v1 → v3 三轮迭代中 agent 看到"Step 3"会先想到"替换"，但口语词精简（v1 漏跑的 Step 3.6）藏在 `Step 3.6` 这个小数后缀里，agent 容易跳过。

**决策**

重构为 **5 个 Phase + 子步骤**：

| Phase | Step | 性质 | 默认 |
|-------|------|------|------|
| 1 准备 | 1.1 读取配置 / 1.2 TXT→MD | 不接触文本 | — |
| 2 识别 | 2.1 文件结构 / 2.2 通读误转写 | 只看不改 | — |
| 3 修正 | 3.1 词典替换（改字） / 3.2 空白清理（改空白） / 3.3 口语词精简（删字） | 改三类不同对象 | **开启** |
| 4 润色 | 4.1 发言人合并 / 4.2 标点规范 / 4.3 段落切分 | 结构调整 | 关闭 |
| 5 输出 | 5.1 主归档 / 5.2 镜像 | 与前 4 步解耦 | — |

**理由**
- "准备 / 识别 / 修正 / 润色 / 输出"五阶段性质分明
- Phase 3 内部三个 Step 改的是三类不同对象（字 / 空白 / 填充词），各自独立可单独审计
- Phase 3 vs Phase 4 区分"读起来对"（无损/轻损，默认开）vs"看起来专业"（结构调整，默认关）
- 整数编号不再混用小数后缀，避免 v1 那种"漏跑 Step 3.6"重演

**影响**
- `SKILL.md` 整体重写工作流章节
- `references/correction_patterns.md` 全面同步 step 引用
- `version: 1.0.4` → `1.0.5`
- 已有的 `correction_log.md` / `meta.json` 不需要回溯改名（保留"## 高置信替换" / "## 基础空白清理" / "## 口语词精简" 三个小节标题，因为 grep 必检是按标题字符串匹配）

**用户决策**
